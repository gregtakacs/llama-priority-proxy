# llama-priority-proxy

An OpenAI-compatible request routing proxy for [llama.cpp](https://github.com/ggml-org/llama.cpp) that sits in front of multiple `llama-server` child processes, managing VRAM-aware model loading, priority-based scenario switching, capacity queuing, and idle eviction — all through a single port.

## Features

- **Priority-based scenario switching** — Define named groups of models (scenarios) with priorities. Requests are routed to the right scenario; lower-priority ones are preempted.
- **VRAM-aware sizing** — Models are loaded at computed context sizes based on actual GPU VRAM capacity, not guesswork.
- **Capacity queuing** — When a model's slots are full, requests queue with SSE heartbeat keep-alive instead of failing immediately.
- **Idle eviction** — Non-default scenarios automatically revert to the default scenario after their keep_alive timeout.
- **Fallback model aliasing** — Tag a resident model to cover another scenario's role, avoiding unnecessary switches.
- **Eager-load pinned models** — Mark a model with `keep_alive: -1` to load it immediately when its scenario activates.
- **OpenAI-compatible API** — Works with any OpenAI-compatible client (Open WebUI, etc.) via `/v1/chat/completions`, `/v1/embeddings`, etc.
- **Bearer token authentication** — Single boundary at the proxy edge; child processes bind to `127.0.0.1` only.
- **GPU VRAM registry** — Per-model VRAM baselines from benchmarking eliminate guesswork on what fits.

## Architecture

```
Client → :11444 (this proxy)
             ├── /v1/chat/completions  →  scenario model (e.g. :11446)
             ├── /v1/embeddings        →  standalone model (e.g. :11445)
             ├── /v1/models            →  list all available labels
             ├── /status               →  current state (scenarios, GPU, loaded models)
             └── /health               →  health check (no auth required)
```

The proxy is a Python aiohttp application. It spawns `llama-server` child processes via subprocess, monitors their health, and routes requests to them over HTTP on `127.0.0.1` (they're never exposed externally).

## Prerequisites

- **Python 3.10+** (for Docker or direct run)
- **NVIDIA GPU** with CUDA support
- **llama-server** binary built with CUDA (the Docker image includes this)
- **Model GGUF files** placed in a models directory (e.g., `/models` inside the container)
- **aiohttp** (`pip install aiohttp` if running outside Docker)

## Configuration

### Overview

The `config/` directory contains all routing and model configuration. **Several files in this directory are auto-generated by benchmarking/inspecting tools** (see below). They are environment-specific and should be created after cloning the repo.

### Configuration Files

| File | Generated by | Description |
|------|-------------|-------------|
| `model_options.json` | `inspect --write-options` | Per-model options: parallel slots, context ranges, GPU offload, labels, keep_alive |
| `model_vram_registry.json` | `bench` | Per-model VRAM baselines from benchmarking |
| `scenario_*.json` | Manual | Named model groups with priority, default model, ports, fallback aliases |
| `standalone_models.json` | Manual | Models always resident outside the scenario system (e.g., embeddings) |

### How to Generate Config Files

1. **Place your GGUF model files** in your models directory.

2. **Set the number of concurrent processes (`parallel`) for each model** before benchmarking — this is critical because it directly determines VRAM baseline (more concurrent slots = larger KV cache = more compute buffers = higher VRAM). The `parallel` value must be set correctly here; it is **baked into the benchmark** via a config signature stored alongside each registry entry, so changing it after `bench` without re-running produces stale data and wrong `solve` results. A good rule of thumb: set `parallel` to the number of concurrent requests you actually expect to serve — not an arbitrary maximum. The inspector creates each model with a default `parallel: 1`.

   **How to set `parallel`:**
   - Open `config/model_options.json` after running `inspect`
   - Find your model under `"options"` and set `"parallel": N` where `N` is the number of concurrent slots
   - Save the file — `bench` will use whatever `parallel` value is set

   Example:
   ```json
   {
     "options": {
       "Qwen3.6-35B-A3B-UD-Q4_K_XL": {
         "parallel": 4,
         "label": "Qwen3.6-35B-A3B-UD-Q4_K_XL",
         "keep_alive": ""
       }
     }
   }
   ```

3. **Benchmark VRAM** to populate `model_vram_registry.json`:
   ```bash
   # Docker backend (default — spins up its own llama-server container per trial):
   python benchmark_vram.py bench --models-dir /path/to/models --image llama-cpp-priority-proxy
   ```
   This measures actual VRAM usage for each model at two context sizes, fits a linear model, and solves for maximum context given your VRAM budget. Use `--backend native` if `llama-server` is already on your `PATH`. The `--model` flag benchmarks a single model; `--force` re-measures even if already present.

4. **Create scenario files** (`config/scenario_chat.json`, etc.) defining your model groups. The proxy evaluates VRAM constraints live during startup, so you can use `"ctx": "auto"` to let it determine the right context size based on the VRAM registry:
   ```json
   {
     "name": "chat",
     "priority": 1,
     "default": true,
     "port": 11446,
     "models": [
       {
         "name": "Qwen3.6-35B-A3B-UD-Q4_K_XL",
         "ctx": "auto",
         "slot": "primary",
         "label": "TakacsAI",
         "fallback_for": ["coding"]
       }
     ]
   }
   ```

5. **Create `config/standalone_models.json`** for models that should always be loaded (e.g., embeddings):
   ```json
   {
     "models": [
       {
         "name": "nomic-embed-text",
         "port": 11445,
         "ctx": 8192
       }
     ]
   }
   ```

### Optional: Pre-computing Context Sizes

Before running the proxy, you can optionally use the `solve` command to pre-compute the maximum context each model can use within your VRAM budget. This is useful for planning and documentation — the proxy will still evaluate constraints live at runtime:

```bash
# Single model — finds the max ctx-size that fits:
python benchmark_vram.py solve --model Qwen3.6-35B-A3B-UD-Q4_K_XL --budget-gb 31

# Multi-model scenario — one model is 'auto', others fixed:
python benchmark_vram.py solve --scenario config/scenario_coding.json --budget-gb 31
```

## Deployment

### Docker (Recommended)

The Docker image is the recommended deployment method — it bundles a custom-built llama.cpp binary compiled with CUDA support for NVIDIA GPUs (targeting Blackwell/RTX 5090 architecture, `sm_120`).

**Build the image:**
```bash
docker build -f Dockerfile.llama-cpp-priority-proxy -t llama-priority-proxy .
```

**Run a container** (adjust paths for your environment):
```bash
docker run -d \
  --name llama-cpp \
  --gpus all \
  -p 11444:11444 \
  -e PROXY_API_KEY_FILE=/run/secrets/proxy_api_key \
  -v /path/to/models:/models:ro \
  -v /path/to/config:/app/config:ro \
  --restart unless-stopped \
  llama-priority-proxy
```

#### Docker Compose Example

A ready-to-adapt example based on a NAS-STACK setup:

```yaml
services:
  llama-cpp:
    container_name: llama-cpp
    build:
      context: .
      dockerfile: Dockerfile.llama-cpp-priority-proxy
    runtime: nvidia
    restart: unless-stopped
    network_mode: host
    environment:
      - NVIDIA_VISIBLE_DEVICES=all
      - PROXY_API_KEY_FILE=/run/secrets/proxy_api_key
    volumes:
      - /path/to/models:/models:ro
      - /path/to/config:/app/config:ro
    secrets:
      - source: ollama_token
        target: /run/secrets/proxy_api_key
```

> **Note:** `network_mode: host` is recommended to avoid port mapping conflicts when scenarios use multiple ports on `127.0.0.1`.

### Running Directly (Development)

You can run the proxy directly with Python without Docker. This requires a locally-built `llama-server` binary with CUDA support. **Docker is recommended for production** since it handles the llama.cpp build and CUDA setup automatically.

```bash
pip install aiohttp
python llama_priority_proxy.py --models-dir /path/to/models --config-dir config --port 11444
```

**Required CLI flags:**

| Flag | Default | Description |
|------|---------|-------------|
| `--models-dir` | `/models` | Directory containing GGUF model files |
| `--config-dir` | `config` | Directory containing configuration JSON files |
| `--port` | `11444` | Port to listen on |
| `--gpu-index` | `0` | Index of the GPU to use |
| `--headroom-gb` | `1.0` | GB of VRAM to reserve as headroom |
| `--load-timeout` | `300` | Seconds to wait for a model to become healthy |
| `--max-wait` | `60` | Max seconds to wait for a free slot before returning an error |

**Environment variables** (override CLI defaults):

| Variable | Default | Description |
|----------|---------|-------------|
| `PROXY_API_KEY` | (unset) | Bearer token for API authentication |
| `PROXY_API_KEY_FILE` | (unset) | Path to file containing the API key (Docker secrets) |
| `PROXY_HEADROOM_GB` | `1.0` | VRAM headroom in GB |
| `PROXY_LOAD_TIMEOUT_S` | `300` | Model load timeout in seconds |
| `PROXY_MAX_WAIT_S` | `60` | Max wait time for a free slot |

## API Endpoints

| Endpoint | Method | Auth Required | Description |
|----------|--------|---------------|-------------|
| `/v1/chat/completions` | POST | Yes | Chat completions (primary endpoint) |
| `/v1/completions` | POST | Yes | Text completions |
| `/v1/embeddings` | POST | Yes | Embeddings |
| `/v1/models` | GET | Yes | List available model labels |
| `/status` | GET | Yes | Current proxy state (GPU, scenarios, loaded models) |
| `/health` | GET | No | Health check (Docker HEALTHCHECK) |

### Authentication

Set `PROXY_API_KEY` (or `PROXY_API_KEY_FILE` pointing to a file containing the key). Requests must include:
```
Authorization: Bearer <your-api-key>
```

The `/health` endpoint is always exempt from authentication checks.
